# 同時注文処理（排他制御）テストシナリオ設計書

## 1. テストの目的
複数ユーザー（複数スレッド）から同時に同一商品の注文処理が実行された場合においても、データの不整合（在庫数のマイナスや、在庫数以上の注文完了）が発生しないこと、すなわちトランザクション制御と **悲観的ロック（`SELECT FOR UPDATE`）** が正しく機能していることを検証する。

## 2. 前提条件（テストデータの準備）
- **対象商品**: テスト用の商品およびインベントリデータ（例: 商品ID `1` ）
- **初期在庫数**: `Inventory` テーブルにて在庫数 `1` 個（最も境界値として不具合が起きやすい状態を設定）
- **注文数**: `OrderRequest` にて各リクエストにつき `1` 個
- **状況**: `ExecutorService` などを利用したJUnitテストコード。複数のスレッドから同時にアクセスを発生させる。

## 3. 実行手順（テストシナリオ）
1. **データ準備**: DBへ対象商品と対象インベントリ（在庫数=1）をテストメソッド内でセットアップする。
2. **並行処理の準備**: Javaの `ExecutorService` などを利用して、同時に実行するスレッドを複数（例: 10スレッド）準備する。
3. **タイミングの同期**: `CountDownLatch` 等を利用し、スレッドがばらばらに処理を始めないよう完全に待機させる。
4. **一斉実行**: ロックを解放し、全10スレッドから同時に `OrderService.createOrder()` メソッドを呼び出す。
5. **待機**: 全てのスレッドの処理が完了するまで待機し、発生した例外を収集する。

## 4. 期待される結果（アサーション・確認項目）
テスト処理完了後、システムが以下の状態になっていることを確認する。

1. **成功件数の確認**:
   - 同時実行した10スレッドのうち、正常終了（戻り値として `OrderResponse` を返却）したのは **「1件のみ」** であること。
2. **失敗（排他エラー）件数の確認**:
   - 残りの **「9件」** のスレッドは、`Inventory` 取得時の悲観的ロック待ち合わせ後に在庫不足と判定され、独自の例外である **`OutOfStockException`** がスローされること。
3. **データ整合性の確認（DBの最終状態）**:
   - 対象商品の最終的な `Inventory` の在庫数が **「0」** であり、マイナス値になっていないこと。
   - `OrderRepository` に保存された対象の新規注文レコードが **「1件のみ」** であること。
